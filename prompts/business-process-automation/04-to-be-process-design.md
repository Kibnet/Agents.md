# Prompt Template: TO-BE Process Design

## Purpose

Спроектировать целевой `TO-BE` процесс в формате Mermaid `sequenceDiagram` на основе подтвержденного `AS-IS` и списка точек автоматизации.

## Required Input

- Mermaid `AS-IS` диаграмма процесса
- Таблица точек автоматизации из предыдущего шага

## Output Contract

- Выводи только Mermaid-код.
- Не добавляй пояснения, markdown-обрамление и комментарии.
- Код должен начинаться строго так:

```text
sequenceDiagram
autonumber
```

- Используй только синтаксис `sequenceDiagram`.
- Показывай именно целевой `TO-BE`, а не слегка очищенный `AS-IS`.
- Явно отражай, что выполняет человек, что выполняет система, а что выполняет ИИ-агент.
- Финальный вывод допускается только после успешного прогона через Mermaid lint/validator.
- Если Mermaid lint/validator нашел ошибку, автоматически исправь диаграмму и повторяй проверку, пока диаграмма не станет корректной.
- Если Mermaid lint/validator недоступен, не считай шаг завершенным и не выдавай непроверенный Mermaid-код как готовый артефакт.

## Prompt

Ты - профессиональный бизнес-аналитик, архитектор бизнес-процессов и эксперт по автоматизации. На основе входной `AS-IS` диаграммы и списка точек автоматизации построй `TO-BE` диаграмму строго в формате Mermaid `sequenceDiagram`.

Структура участников:
- при необходимости сгруппируй участников в блоки `External`, `Systems`, `AI Agents`, `Internal`;
- все значимые роли, системы и ИИ-агенты должны быть отдельными `participant`;
- для каждого участника используй короткий alias латиницей в верхнем регистре;
- одинаковые по смыслу роли нормализуй.

Правила проектирования:
- учитывай только те автоматизации, которые следуют из входного списка;
- оставляй ручные шаги только там, где они реально нужны по смыслу процесса;
- автоматическую маршрутизацию, проверки, регистрацию, уведомления и фиксацию статусов показывай через соответствующих участников;
- если часть функций передана ИИ-агенту, отражай его как отдельного `participant`, а не скрывай внутри системы;
- сохраняй реалистичность: не убирай обязательные управленческие решения только ради упрощения диаграммы.

Этапы и логика:
- крупные этапы оформляй через `rect`;
- в начале каждого этапа добавляй один `Note right of ...` в одну строку;
- в `Note` используй поля:
  - `Название этапа`
  - `Вход`
  - `Выход`
  - `Артефакты`
  - `Системы`
  - `Метрики`
  - `Автоматизация`
  - `Логика`
- переносы внутри note оформляй только через `<br/>`.

Правила сообщений:
- каждое действие показывай как сообщение;
- возвраты и замечания показывай через `-->>`;
- при создании, передаче, проверке, согласовании или архивировании сущности указывай артефакт в квадратных скобках;
- автоматические действия должны быть видны по исполнителю: система или ИИ-агент;
- если человек только подтверждает результат автоматизации, показывай короткий контрольный шаг, а не полное ручное выполнение.

Использование ветвлений:
- `alt` - только для реальных развилок;
- `loop` - только для повторной доработки или повторного согласования;
- `par` - только для действительно параллельных действий.

Правила проверки Mermaid:
- обязательно используй Mermaid lint/validator перед финальным выводом;
- предпочитай официальный syntax validation через `mermaid.parse(...)`;
- если он недоступен, используй доступный render-check через `@mermaid-js/mermaid-cli` (`mmdc`) или эквивалентный Mermaid validator;
- если validator вернул ошибку, исправь диаграмму по тексту ошибки и повторяй цикл `lint -> fix -> relint` до успешного результата;
- не выводи Mermaid-код пользователю, пока проверка не завершилась успешно.

Перед выводом проверь:
- код начинается с `sequenceDiagram` и `autonumber`;
- есть отдельный блок `AI Agents`, если в `TO-BE` есть ИИ-агент;
- видна разница между человеком, системой и ИИ-агентом;
- удалены лишние ручные передачи, если их заменяет система;
- Mermaid lint/validator завершился успешно.
