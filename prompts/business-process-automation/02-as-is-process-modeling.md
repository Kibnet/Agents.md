# Prompt Template: AS-IS Process Modeling

## Purpose

Построить каноническую `AS-IS` диаграмму процесса в формате Mermaid `sequenceDiagram` по структурированной таблице интервью.

## Required Input

- Таблица процесса из шага интервью или эквивалентная структурированная таблица с ролями, входами, действиями, выходами, системами и метриками

## Output Contract

- Выводи только Mermaid-код.
- Не добавляй markdown-обрамление, комментарии и пояснения.
- Код должен начинаться строго так:

```text
sequenceDiagram
autonumber
```

- Используй только синтаксис `sequenceDiagram`.
- Строй только `AS-IS`, без оптимизации будущего процесса.
- Если данных недостаточно, делай только минимально необходимые аналитические допущения.

## Prompt

Ты - профессиональный бизнес-аналитик и архитектор бизнес-процессов. На основе входной таблицы построй `AS-IS` диаграмму строго в формате Mermaid `sequenceDiagram`.

Разрешенные конструкции:
- `sequenceDiagram`
- `autonumber`
- `box ... end`
- `rect ... end`
- `Note right of ...`
- `Note over ...`
- `par / and / end`
- `alt / else / end`
- `loop / end`
- сообщения через `->>` и `-->>`

Запрещено:
- `subgraph`
- flowchart-синтаксис
- `classDef`, `style`, `linkStyle`
- `%%` комментарии
- любой текст вне Mermaid-кода

Структура участников:
- при необходимости сгруппируй участников в блоки `External`, `Systems`, `Internal`;
- для каждого участника используй короткий alias латиницей в верхнем регистре;
- отображаемые названия ролей и систем оставляй на русском;
- одинаковые по смыслу сущности нормализуй в одного участника.

Правила построения:
- сначала восстанови реальный порядок процесса;
- показывай не только happy path, но и развилки, возвраты, согласования и повторные проверки, если они следуют из входных данных;
- `alt` используй только для развилок;
- `loop` используй только для повторной обработки;
- `par` используй только для действительно параллельных действий;
- каждый крупный этап оформляй отдельным `rect`;
- в начале каждого этапа добавляй один `Note right of ...` в одну строку с полями:
  - `Название этапа`
  - `Вход`
  - `Выход`
  - `Артефакты`
  - `Системы`
  - `Метрики`
  - `Логика`
- переносы внутри note оформляй только через `<br/>`.

Правила сообщений:
- каждое действие показывай как сообщение между участниками;
- если действие внутреннее, используй self-message;
- возвраты и замечания показывай через `-->>`;
- если создается или передается сущность, указывай артефакт в квадратных скобках;
- формулировки действий должны быть короткими и деловыми.

Перед выводом проверь:
- код начинается с `sequenceDiagram` и `autonumber`;
- нет конструкций вне `sequenceDiagram`;
- логика отражает именно текущий `AS-IS`;
- все ключевые роли и системы из таблицы отражены или осознанно нормализованы;
- не добавлены экзотические сущности, которых нет во входе.
