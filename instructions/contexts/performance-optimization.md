# Context: Performance Optimization

## Когда применять

- Нужно найти bottleneck/hot path и предложить оптимизацию.
- Нужно провести benchmark-driven цикл улучшений с измеримым эффектом.
- Нужно оформить решение с оценкой рисков и результатами до/после.

## Когда не применять

- Для обычных функциональных задач без требований по производительности.
- Для изменений, где измерение эффекта невозможно или нерелевантно.

## MUST

- Для каждого optimization item фиксировать гипотезу, способ замера, риски и тесты.
- Снимать baseline до изменения и post-замер после изменения в одинаковой конфигурации.
- До и после изменений запускать тесты, начиная с targeted и заканчивая full-run.
- Оставлять только те изменения, где прирост значим и поддерживаемость не деградирует.
- Каждую независимую оптимизацию коммитить отдельно.

## SHOULD

- Сравнивать минимум две метрики: latency (`Mean`) и allocations (`Allocated`).
- При шумных результатах повторять замер до устойчивого тренда.
- В отчете по пункту фиксировать решение: `оставлено` или `откачено` с причиной.

## MAY

- Использовать дополнительные профайлеры, если benchmark не дает достаточной детализации.
- Переформулировать гипотезу и повторить цикл для того же hot path.

## Команды

```powershell
# Примерная схема
# 1) baseline benchmark
# 2) targeted tests
# 3) change
# 4) targeted + full tests
# 5) post benchmark
```
