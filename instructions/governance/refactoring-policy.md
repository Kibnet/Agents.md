# Governance: Refactoring Policy

## Когда применять

- Для любого рефакторинга кода: локального, архитектурного или механического.
- Когда меняется внутренняя структура без намеренного изменения внешнего поведения.
- Когда производительность улучшается через структурные изменения или локализованную оптимизацию.

## Когда не применять

- Для чистого функционального изменения без структурного рефакторинга.
- Для эстетических правок, которые не снижают риск, связанность или стоимость изменений.
- Для performance-оптимизации без измерений, если ради неё ухудшается понятность кода.

## MUST

- До существенных структурных изменений понять фактическое текущее поведение, границы задачи, скрытые зависимости, инварианты и точки побочных эффектов.
- До существенных структурных изменений построить минимальную страховку: существующие тесты, characterization tests, contract checks или другой воспроизводимый safety net.
- Явно отделять структурный refactor от функциональных изменений и не маскировать новое поведение под `refactor`.
- Предпочитать снижение когнитивной сложности, связанности и объёма скрытого знания перед добавлением абстракций "на будущее".
- Любой performance tradeoff, ухудшающий понятность, подтверждать измерением до и после на релевантном пути выполнения.
- Выполнять refactor малыми проверяемыми шагами, которые можно объяснить по diff и локально откатить.
- После refactor обновлять или удалять устаревшие комментарии, docstring и документацию.

## SHOULD

- Предпочитать удаление лишнего слоя, дублирования и мёртвого кода добавлению новых сущностей.
- Разделять чистую логику и side effects, локализовать сложность и expensive-операции.
- Держать diff в форме, удобной для поэтапного review: переименование, выделение, перенос, удаление, оптимизация.
- Сохранять или улучшать наблюдаемость: логи, метрики, трассировку и диагностируемость ошибок.

## MAY

- Фиксировать baseline-замеры, rollback-план и матрицу неизменяемых контрактов прямо в spec или PR-описании.
- Добавлять краткую refactor-сводку для reviewers, если diff затрагивает несколько последовательных шагов.

## Команды

```text
git diff --stat
dotnet test
npm test
```
