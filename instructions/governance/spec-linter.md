# Governance: SPEC Linter

## Когда применять

- Перед подтверждением и реализацией новых `spec`-документов.
- При ревью или аудите качества `specs`-документации.

## Когда не применять

- Для задач без `SPEC`-гейта.
- Для запросов без изменений файлов и без фазы реализации.

## MUST

- Проверить все пункты `1..20` по блокам `A..F`.
- Для каждого пункта выставить `PASS` / `PARTIAL` / `FAIL` и короткий комментарий.
- Итог считать по правилу:
  - `ГОТОВО`, если нет `FAIL` в блоках `A`, `C`, `D`.
  - `НУЖНА ДОРАБОТКА`, если есть хотя бы один `FAIL` в блоках `A`, `C`, `D`.
- Результат линтера сохранять в самой спецификации.

## SHOULD

- Для `PARTIAL` явно фиксировать, что именно не хватает.
- Проверять полноту секций `Команды` и `Открытые вопросы` перед фазой `EXEC`.
- Держать комментарии короткими и проверяемыми (факт, риск, действие).

## MAY

- Добавлять блок `G` с проектными проверками.
- Вести отдельный audit-файл для крупных `large` спецификаций.

## Команды

```markdown
### SPEC Linter Result

| Блок | Пункты | Статус | Комментарий |
|---|---|---|---|
| A. Полнота спеки | 1-5 | PASS/PARTIAL/FAIL | ... |
| B. Качество дизайна | 6-10 | PASS/PARTIAL/FAIL | ... |
| C. Безопасность изменений | 11-13 | PASS/PARTIAL/FAIL | ... |
| D. Проверяемость | 14-16 | PASS/PARTIAL/FAIL | ... |
| E. Готовность к автономной реализации | 17-19 | PASS/PARTIAL/FAIL | ... |
| F. Соответствие профилю | 20 | PASS/PARTIAL/FAIL | ... |

Итог: ГОТОВО / НУЖНА ДОРАБОТКА
```

## Быстрый чек-лист блоков

| Блок | Что обязательно проверить |
|---|---|
| A | Цель, AS-IS, проблема, цели дизайна, Non-Goals |
| B | Ответственность, интеграция, бизнес-правила, ошибки, perf |
| C | Данные, миграция, совместимость, rollback |
| D | Acceptance Criteria, тест-план, команды проверки |
| E | План этапов, отсутствие блокирующих вопросов, масштаб |
| F | Соответствие выбранному profile |
```
