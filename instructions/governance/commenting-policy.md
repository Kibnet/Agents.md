# Governance: Commenting Policy

## Когда применять

- Когда задача включает целенаправленное массовое комментирование, cleanup комментариев или review существующих комментариев.
- Когда код содержит скрытые межфайловые связи, инварианты, side effects, workaround или ограничения внешних систем, которые нельзя надёжно восстановить простым чтением файла.
- Когда нужно унифицировать правила docstring / documentation comments для публичных API.

## Когда не применять

- Когда смысл кода уже очевиден из имён, типов, тестов и структуры.
- Когда проблему лучше устранить неймингом, выделением функции, упрощением логики, усилением типов или тестом.
- Когда комментарий только пересказывает код, историю изменений или субъективные замечания.

## MUST

- Сначала пытаться убрать необходимость в комментарии структурой кода: именами, рефакторингом, типами и тестами.
- Добавлять комментарий только там, где без него будущий читатель с высокой вероятностью сделает неверный вывод о коде.
- Комментировать скрытые межфайловые связи, неочевидные инварианты, побочные эффекты, workaround, доменные ограничения и участки с высокой ценой неправильного изменения.
- Если комментарий описывает внешнюю связь, явно указывать:
  - где находится связанная логика;
  - как она влияет на текущий код;
  - что опасно менять независимо и при каком условии изменение безопасно.
- Удалять или переписывать комментарии, которые стали ложными, неполными, устаревшими или больше не экономят время понимания.
- Для публичного API использовать docstring / documentation comments для описания контракта: назначения, предусловий, постусловий, side effects и семантики ошибок, а не деталей реализации.

## SHOULD

- Писать про `почему` и `из-за чего`, а не про `что`.
- Называть конкретные сущности: путь к файлу, модуль, событие, флаг, поле, таблицу, индекс, ключ кэша.
- Держать комментарий коротким, но завершённым: один точный абзац лучше нескольких расплывчатых строк.
- Размещать комментарий там, где у читателя возникает вопрос: над блоком, рядом с выражением или в doc-comment контракта.
- При массовом улучшении комментариев сначала удалять мусорные комментарии, а затем комментировать только hotspot-участки с высокой ценой непонимания.

## MAY

- Использовать формулу `причина -> внешняя связь/ограничение -> следствие -> условие безопасного изменения`.
- Оставлять в комментарии краткую ссылку на связанный файл, модуль, job, migration, индекс или внешний контракт, если это ускоряет безопасное изменение кода.

## Команды

```powershell
# Найти слабые комментарии и маркеры, которые требуют ручной проверки
rg -n "TODO|FIXME|магия|костыль|не трогать" .

# Просмотреть комментарии и doc-comments в hotspot-области
rg -n "^\s*(//|///|/\*|\*)" .
```
