# Governance: SPEC Rubric

## Когда применять

- Для оценки готовности `spec` к автономной реализации.
- Для сравнения качества спецификаций разных объёмов (`small/medium/large`).

## Когда не применять

- Для задач без формализованной спецификации.
- Когда достаточно устного согласования и не требуется фиксированная метрика.

## MUST

- Оценить все 6 критериев по шкале `0/2/5`.
- Суммировать балл и определить зону:
  - `0-10`: рискованно;
  - `11-20`: под контролем;
  - `21-30`: готово к автономному выполнению.
- Фиксировать итоговый балл и слабые места в самой спецификации.
- Использовать порог `21` как ориентир для автономного запуска.

## SHOULD

- Давать короткое обоснование каждому баллу.
- Для критичных задач повышать проходной порог и указывать это явно.

## MAY

- Добавлять дополнительную пометку по уровню риска внедрения.
- Подкручивать шкалу под доменные требования команды.
- Разделять итог на `критическая / высокая / средняя` зону риска по бизнес-важности.

## Команды

```markdown
### SPEC Rubric Result

| Критерий | Балл (0/2/5) | Обоснование |
|---|---|---|
| 1. Ясность цели и границ | | |
| 2. Понимание текущего состояния | | |
| 3. Конкретность целевого дизайна | | |
| 4. Безопасность (миграция, откат) | | |
| 5. Тестируемость | | |
| 6. Готовность к автономной реализации | | |

Итоговый балл: __ / 30
Зона: рискованно / под контролем / готово к автономному выполнению
```

## Шкала 0/2/5

| Балл | Интерпретация |
|---|---|
| 0 | Пункт отсутствует или непроверяем |
| 2 | Пункт частично раскрыт, есть неопределённость |
| 5 | Пункт раскрыт полно и проверяемо |
```
