# Profile: Product System Design

## Когда применять

- Для проектирования новых подсистем, библиотек или сервисных модулей.
- Для задач с явно выраженными требованиями к API, безопасности и публичным интерфейсам.

## Когда не применять

- Для точечных багфиксов в уже существующей, стабилизированной подсистеме.
- Для чисто косметических изменений без влияния на архитектуру.

## MUST

- Чётко выделять `цели` и `non-goals`.
- Описывать целевую архитектуру и границы подсистемы.
- Зафиксировать публичный API и договоры совместимости.
- Проверять аспекты безопасности и конфигурации интеграций.

## SHOULD

- Подтягивать ADR/контрактные решения из существующей архитектуры.
- Описывать UX/конфигурационные сценарии, где это влияет на поведение системы.

## MAY

- Добавлять альтернативные пути реализации и мотивацию выбора.
