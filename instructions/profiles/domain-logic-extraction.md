# Profile: Domain Logic Extraction

## Когда применять

- Для вынесения бизнес-правил из UI/VM слоёв в отдельный доменный слой.
- Для задач по формализации и централизации бизнес-логики.

## Когда не применять

- Для локальных правок без изменения доменных правил.
- Если задача касается только инфраструктурной настройки без бизнес-логики.

## MUST

- Формализовать доменные правила и их источники.
- Определить матрицу триггеров и сценариев обработки.
- Зафиксировать тесты на правила и их граничные случаи.
- Сохранять согласованность между источниками данных и применением правил.

## SHOULD

- Выделять правила в отдельные модули с минимальными зависимостями.
- Поддерживать покрытие тестами `unit` + `property` для ключевой логики.

## MAY

- Добавлять диагностические метрики для проверки частых конфликтов бизнес-правил.

## Команды

```text
dotnet test
python -m pytest
```
