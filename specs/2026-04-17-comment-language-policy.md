# Нормализация языка комментариев по конвенции репозитория

## 0. Метаданные
- Тип (профиль): `product-system-design`
- Владелец: `instructions/governance/commenting-policy.md`
- Масштаб: small
- Целевой релиз / ветка: `2.1.1` / текущая рабочая ветка
- Ограничения:
  - не размазывать одно и то же правило по нескольким owner-документам;
  - не менять routing и conflict model без необходимости;
  - не ослаблять модель `AGENTS.override.md` как локального ужесточения центральных MUST;
  - ограничить изменение минимальным набором документов.
- Связанные ссылки:
  - `instructions/governance/routing-matrix.md`
  - `instructions/governance/commenting-policy.md`
  - `instructions/onboarding/AGENTS.override.template.md`
  - `instructions/governance/document-contract.md`
  - `instructions/governance/versioning-policy.md`
  - `CHANGELOG.md`

## 1. Overview / Цель
Добавить в каталог инструкций явное правило выбора языка комментариев в коде для задач, связанных с добавлением, переписыванием или массовым cleanup комментариев. Правило должно приводить новые и изменяемые комментарии к языковой конвенции конкретного репозитория и не допускать случайного смешения языков. Изменение нужно разместить в одном owner-документе и дополнить локальным onboarding-примером без лишнего дублирования.

## 2. Текущее состояние (AS-IS)
- `instructions/governance/commenting-policy.md` уже является owner-документом для полезных комментариев, cleanup комментариев и doc-comments.
- Текущая `commenting-policy` регулирует смысл, плотность и допустимость комментариев, но не задаёт явное правило выбора языка комментариев.
- `instructions/core/collaboration-baseline.md` требует вести рабочее общение с пользователем на русском языке, но это правило относится к диалогу агента, а не к комментариям в коде.
- `instructions/governance/document-contract.md` фиксирует русский язык контента для `instructions/*`, но это правило относится к каталогу инструкций, а не к потребительским репозиториям.
- `instructions/onboarding/AGENTS.override.template.md` допускает локальные project-specific MUST, но сейчас не показывает пример repo-specific фиксации языка комментариев.
- Из-за отсутствия отдельного языкового правила агент при массовом комментировании может ориентироваться на локальный фрагмент кода или на язык текущего контекста и иногда оставлять комментарии на другом языке, чем принято в репозитории.

## 3. Проблема
В каталоге инструкций нет канонического правила, которое обязывает сначала определить принятый язык комментариев в текущем репозитории и затем приводить к нему все новые и изменяемые комментарии, из-за чего при массовом комментировании возможен дрейф языка и смешение комментариев.

## 4. Цели дизайна
- Зафиксировать единый owner-документ для правила языка комментариев.
- Сделать правило детерминированным: чтобы агент понимал, как выбрать язык автоматически и когда опираться на явное указание.
- Сохранить минимальность каталога без копирования одного и того же требования в baseline, routing и review-документы.
- Сохранить совместимость с моделью `AGENTS.override.md` как локального repo-specific ужесточения.
- Не смешивать язык рабочего диалога агента и язык комментариев в коде.

## 5. Non-Goals (чего НЕ делаем)
- Не вводим глобальное требование писать комментарии во всех репозиториях только на одном языке.
- Не переписываем все существующие комментарии во всех репозиториях автоматически.
- Не меняем `routing-matrix.md`, если существующая маршрутизация уже указывает owner-документ корректно.
- Не дублируем языковое правило в `collaboration-baseline.md`, `quest-mode.md` или `review-loops.md`, если это не требуется для работоспособности.
- Не меняем язык самих документов `instructions/*`: для них продолжает действовать `document-contract.md`.

## 6. Предлагаемое решение (TO-BE)
### 6.1 Распределение ответственности
- `instructions/governance/commenting-policy.md` -> каноническое правило выбора и нормализации языка комментариев.
- `instructions/onboarding/AGENTS.override.template.md` -> пример локального repo-specific MUST для явной фиксации языка комментариев в потребительском репозитории.
- `CHANGELOG.md` -> отражение изменения как patch-level уточнения контракта каталога.

### 6.2 Детальный дизайн
- В `commenting-policy.md` добавить короткий блок `MUST`, который требует перед добавлением, переписыванием или массовым cleanup комментариев определить принятый язык комментариев для текущего репозитория.
- Язык определять по явному и воспроизводимому приоритету:
  - repo-specific правило из локального `AGENTS.override.md` или других явных инструкций репозитория;
  - явное указание пользователя, если оно не конфликтует с repo-specific правилом;
  - автоматическое определение по преобладающему языку существующих комментариев, docstring и documentation comments.
- Все новые и изменяемые комментарии, docstring и documentation comments приводить к выбранному языку; не сохранять смешение языков по инерции.
- Если явного repo-specific правила нет и в репозитории нет доминирующего языка комментариев, перед массовым комментированием запросить выбор пользователя.
- В `AGENTS.override.template.md` добавить один пример локального MUST, который жёстко закрепляет язык комментариев для конкретного репозитория.
- `routing-matrix.md` не менять, потому что owner-документ для этой темы уже указан: `commenting-policy.md`.
- `collaboration-baseline.md` не менять, потому что он регулирует язык общения агента с пользователем, а не язык комментариев в коде.
- Изменение версии отразить как `PATCH`, поскольку меняется формулировка и точность контракта без новой conflict model и без breaking-изменения существующей маршрутизации.

## 7. Бизнес-правила / Алгоритмы (если есть)
- Алгоритм выбора языка комментариев:
  - если в репозитории есть явное правило о языке комментариев, использовать его;
  - иначе, если пользователь явно указал язык комментариев, использовать его;
  - иначе определить доминирующий язык по существующим комментариям репозитория;
  - если доминирующий язык не определяется надёжно, запросить решение пользователя перед массовым комментированием.
- Алгоритм нормализации:
  - при добавлении новых комментариев использовать выбранный язык;
  - при переписывании и cleanup существующих комментариев переводить изменяемые комментарии к выбранному языку;
  - не оставлять в пределах одного целевого cleanup-прохода смешение языков, возникшее только из-за инерции исходного текста.

## 8. Точки интеграции и триггеры
- Новое правило срабатывает, когда задача включает:
  - целенаправленное массовое комментирование;
  - cleanup комментариев;
  - переписывание существующих комментариев;
  - унификацию docstring / documentation comments.
- Локальный override используется как repo-specific источник истины в consumer-репозитории.
- Проверка применения правила происходит на этапе проектирования решения и при финальном self-review затронутых комментариев.

## 9. Изменения модели данных / состояния
- Новых данных, persisted-состояния и схем хранения нет.
- Изменяется только нормативный контракт каталога инструкций.

## 10. Миграция / Rollout / Rollback
- Rollout:
  - обновить `commenting-policy.md`;
  - добавить локальный пример в `AGENTS.override.template.md`;
  - зафиксировать изменение в `CHANGELOG.md`.
- Обратная совместимость:
  - существующие consumer-репозитории продолжают работать без локального override;
  - при отсутствии override агент использует автоматическое определение языка по комментариям.
- Rollback:
  - удалить новый языковой блок из `commenting-policy.md`;
  - удалить пример из `AGENTS.override.template.md`;
  - откатить запись в `CHANGELOG.md`.

## 11. Тестирование и критерии приёмки
- Acceptance Criteria:
  - `commenting-policy.md` явно требует определить язык комментариев до массового комментирования и задаёт приоритет выбора языка.
  - Политика явно требует приводить новые и изменяемые комментарии к выбранному языку репозитория.
  - Политика явно описывает поведение при неоднозначной смешанной языковой практике.
  - `AGENTS.override.template.md` показывает минимальный пример локальной фиксации языка комментариев без новой дублирующей нормативной секции.
  - `CHANGELOG.md` отражает изменение как patch-level уточнение контракта.
  - Валидатор каталога и тесты валидатора проходят.
- Какие тесты добавить/изменить:
  - новые автоматические тесты не требуются, если validator уже проверяет структуру документов, а изменение не меняет обязательный набор секций.
- Characterization tests / contract checks для текущего поведения (если применимо):
  - ручная проверка, что owner-документом остаётся `commenting-policy.md`, а routing не требует синхронизации.
- Базовые замеры до/после для performance tradeoff (если применимо):
  - не применимо.
- Команды для проверки:
  - `pwsh -File scripts/validate-instructions.ps1`
  - `pwsh -File scripts/test-validate-instructions.ps1`
  - `rg -n "язык комментар|comment language|docstring" instructions`

## 12. Риски и edge cases
- Репозиторий может содержать исторически смешанные комментарии без явного доминирующего языка.
- Пользователь может попросить язык, который конфликтует с локальным repo-specific правилом; в этом случае приоритет должен остаться у локального правила.
- В монорепозитории могут быть подсистемы с разной историей комментариев; базовый контракт остаётся repo-level, а при неустранимой неоднозначности требуется решение пользователя.
- Слишком широкая правка baseline или routing создаст дублирование и усложнит поддержку каталога.

## 18. Альтернативы и компромиссы
- Вариант: добавить правило в `collaboration-baseline.md`.
- Плюсы:
  - правило было бы видно почти во всех задачах.
- Минусы:
  - baseline регулирует язык общения агента, а не owner-правила комментирования;
  - повышается дублирование и размывается ответственность owner-документа.
- Почему выбранное решение лучше в контексте этой задачи:
  - `commenting-policy.md` уже является каноническим owner-документом для комментариев, поэтому новое правило естественно жить именно там.

- Вариант: продублировать правило в `routing-matrix.md`, `review-loops.md` и `commenting-policy.md`.
- Плюсы:
  - правило было бы видно в нескольких точках входа.
- Минусы:
  - дублирование повышает риск расхождения формулировок;
  - для самой маршрутизации это не нужно, owner уже назначен.
- Почему выбранное решение лучше в контексте этой задачи:
  - одна owner-формулировка и один onboarding-пример дают минимальную поддерживаемую поверхность.
