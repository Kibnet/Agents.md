# User Workflow Preferences

Дата подготовки: 2026-06-05.

Источник: устойчивые паттерны из всех доступных Codex-сессий и уже зафиксированные memory guardrails.

## Language And Communication

- Отвечать на русском языке, если пользователь не просит другой язык.
- Для русскоязычных docs/UI использовать грамотный русский без ненужного смешения с английским.
- Технические термины можно оставлять на английском, если это общепринято: `TUnit`, `Playwright`, `NuGet`, `GitHub`, `Avalonia`.
- Финал должен быть коротким, но с evidence: что сделано, что проверено, что не удалось проверить.

## Autonomy

Пользователь часто ожидает автономную работу до результата. Команды вроде "делай", "продолжай", "внеси изменения", "оформи PR", "сделай коммит" обычно означают: не останавливаться на плане, а выполнить очевидный следующий шаг.

Останавливать работу и спрашивать нужно, когда:

- есть несколько жизнеспособных вариантов без явного лучшего;
- изменение затронет публичный API или рискованный product behavior;
- требуется секрет/доступ, которого нет;
- текущая инструкция требует human gate;
- scope стал больше исходной задачи.

## Decision Support

Когда пользователь спрашивает "как предлагаешь", "давай обсудим", "предложи варианты":

1. Дать 2-3 варианта.
2. Для каждого указать tradeoff.
3. Дать рекомендацию.
4. Не превращать это в длинный brainstorm без практического вывода.

## Specs And Review Loops

Пользователь часто использует цикл:

1. Подготовить spec.
2. Провести spec review.
3. Пользователь пишет "Спеку подтверждаю".
4. EXEC.
5. Test/build.
6. Exec review.
7. Commit/PR.

Если в текущем repo instructions включают Quest/SPEC gate, соблюдать его строго.

## Validation Expectations

- На вопрос "а тесты ты прогнал?" ответ должен быть конкретным.
- Для behavior changes нужны tests.
- Для UI changes нужны screenshots/render evidence.
- Для CI failures нужно смотреть job logs, а не гадать.
- Если проверка невозможна, нужно указать blocker и next-best evidence.

## UI Expectations

Пользователь оценивает UI строго:

- compact, dense, no wasted space;
- современный вид controls;
- иконки и кнопки не обрезаны;
- wide и narrow layouts проверены отдельно;
- visual noise убран;
- результат подтверждён screenshots.

Если пользователь просит "до идеала", агент должен сам посмотреть screenshots и честно признать недостатки до финала.

## Delivery Expectations

Для commit/PR:

- проверить git state;
- не включать unrelated changes;
- соблюдать Conventional Commits;
- открыть PR с понятными summary/validation/risks;
- проверить CI или хотя бы указать статус/невозможность проверки;
- не коммитить секреты.

## Artifact Expectations

- Презентации/HTML/PPTX/docs должны быть редактируемыми, если пользователь этого требует.
- Растеризация допустима только как часть assets, а не как замена editable structure.
- Русские документы должны сохранять читаемый текст и корректную кодировку.

## What Annoys The User

- "Готово" без проверок.
- UI, который технически изменён, но визуально плох.
- Неверные тестовые команды.
- Повторение уже найденных ошибок агента.
- Смешение русского и английского без причины.
- Устаревшие пакеты, когда можно использовать latest stable.
- Игнорирование скриншотов.
- Лишние вопросы вместо самостоятельного чтения repo context.

## Practical Assistant Behavior

1. Читать инструкции и repo topology.
2. Понять acceptance criteria.
3. Сделать минимально-достаточное изменение.
4. Запустить правильную targeted проверку.
5. Для UI посмотреть screenshot.
6. Расширить проверку при необходимости.
7. Провести self-review.
8. Кратко отчитаться с evidence.
