# UI Quality Rubric From Sessions

Дата подготовки: 2026-06-05.

Назначение: чеклист самопроверки UI-изменений под устойчивые ожидания пользователя. Использовать для Avalonia, web, презентаций, PDF/document UI и любых visual-facing задач.

## Acceptance Bar

Пользователь не принимает UI как готовый только потому, что:

- проект собрался;
- тест прошёл;
- элемент формально появился;
- screenshot существует, но агент его не оценил.

Минимальный acceptance bar: UI должен быть рабочим, плотным, аккуратным, читаемым, визуально современным и проверенным в релевантных состояниях/размерах.

## Core Principles

### 1. Плотность без тесноты

Ожидание:

- без лишних пустых пространств;
- логичные группы элементов;
- важные controls доступны без скролла, если это реалистично;
- compact layout не ломает читаемость.

Проверить:

- на широком экране элементы, которые должны быть в одну строку, не переносятся;
- на узком экране нет горизонтального overflow;
- группы не создают визуальный шум.

### 2. Кнопки и иконки должны выглядеть законченными

Ожидание:

- иконка полностью видна;
- кнопка имеет стабильный размер;
- icon-only controls понятны по месту и состоянию;
- действия вроде archive/move/delete могут быть объединены в menu, если так компактнее и понятнее.

Проверить:

- иконка не обрезана;
- плюс/шестерёнка/дата/duration controls выглядят одинаково аккуратно;
- hit target не слишком маленький;
- disabled/hover/focus состояние не ломает layout.

### 3. Wide и narrow - разные acceptance states

Ожидание:

- широкая desktop-карточка должна выглядеть не хуже narrow/mobile;
- narrow не должен быть просто "сжатым desktop";
- оба формата требуют отдельной проверки.

Проверить:

- desktop wide screenshot;
- narrow/mobile screenshot;
- переключение responsive states;
- отсутствие "уехало на вторую строку" для элементов, которые должны быть inline.

### 4. Не добавлять визуальный шум

Ожидание:

- названия разделов и подписи нужны только если помогают;
- повторяющиеся labels и избыточные blocks ухудшают UX;
- объединённые compact controls предпочтительнее, если смысл остаётся понятным.

Проверить:

- убрать заголовки, которые не несут decision value;
- заменить длинные кнопки маленькими пиктограммами, если пользователь просит экономить место;
- не дублировать одно и то же значение 3-4 раза.

### 5. Visual evidence обязательно

Ожидание:

- пользователь часто просит "покажи скриншоты для подтверждения";
- агент должен сам смотреть screenshots и честно оценивать результат.

Проверить:

- screenshot после изменений;
- self-review screenshot перед финалом;
- если screenshot невозможен, указать почему и дать next-best evidence.

### 6. User-facing copy должна быть грамотной

Ожидание:

- русский интерфейс и документация должны быть на грамотном русском;
- не смешивать русские и английские слова без необходимости;
- technical English допустим для терминов вроде `TUnit`, `Playwright`, `NuGet`.

Проверить:

- нет mojibake;
- нет случайного английского в русскоязычном UI;
- JSON/data с русским текстом читаемы, если пользователь просит human-readable.

## Scoring Rubric

| Критерий | 0 | 1 | 2 |
|---|---|---|---|
| Layout density | много пустот/переносов | приемлемо, но есть шум | плотный и читаемый |
| Visual polish | элементы обрезаны/кривые | формально работает | аккуратно и современно |
| Responsive states | проверен один размер | проверены два размера частично | wide/narrow проверены и выглядят хорошо |
| Interaction clarity | действия непонятны | понятны после чтения labels | понятны из структуры/иконок/меню |
| Evidence | нет screenshot | screenshot есть, но без self-review | screenshot + явный self-review |
| Tests | нет при behavior change | частично | targeted UI/headless test |

Рекомендация: если сумма ниже 9/12, не завершать UI-задачу без доработки или честного caveat.

## Screenshot Self-Review Prompt

Перед финальным ответом по UI-задаче агент должен ответить себе:

1. Есть ли элементы, которые обрезаны или не полностью видны?
2. Есть ли неожиданный перенос строки?
3. Есть ли лишние заголовки, отступы или повторяющиеся labels?
4. Выглядят ли primary actions доступными без поиска?
5. Сохраняется ли плотность на широком и узком экране?
6. Видно ли состояние, ради которого пользователь просил изменение?
7. Можно ли честно сказать, что результат выглядит лучше, а не просто иначе?

## Red Flags

- "Задача выполнена", но screenshot не посмотрели.
- "Собралось" вместо visual evidence.
- Иконка или плюс частично обрезаны.
- Поля планирования/повторения занимают больше строк, чем нужно.
- Секции плодят визуальный шум.
- Пользователь просит editable artifact, а агент сделал картинку.
- На wide desktop UI хуже, чем на narrow.

## Recommended Final Evidence

Для UI задач финал должен включать:

- изменённые файлы;
- test/build command;
- screenshot path или краткое описание visual check;
- какие viewports/states проверены;
- residual risk, если что-то не проверено.
